## 1. はじめに

[前回（代謝制御編）](02_代謝制御編.md)では、局所混雑度に基づくエッジの適応的減衰を解説しました。混雑した領域を重点的に整理し、疎な領域は慎重に扱う。直感的で合理的なアプローチです。

しかし、代謝制御だけでは根本的な問題が残ります。接続の少ないノードは、たとえ疎な領域で減衰が緩やかであっても、いずれは全てのエッジを失って孤立します。孤立したノードは削除される。

ここに致命的な落とし穴があります。**接続が少ないことは、価値が低いことを意味しません。**

---

## 2. 偽陰性の問題

### 2.1 具体例

以下は、各業界で「接続が少ないが価値が高い」データの例です。

| 業界 | 低接続データ | なぜ価値が高いか |
|---|---|---|
| EC/推薦 | ニッチな専門書 | 特定顧客にとって唯一の選択肢 |
| 医療 | 希少疾患データ | 患者数は少ないが治療に不可欠 |
| 金融 | 不正取引信号 | 接続パターンが異常だからこそ価値がある |
| サイバーセキュリティ | APT（高度持続型脅威）信号 | 痕跡が少ないからこそ見落とせない |
| サプライチェーン | 唯一供給元 | 代替がないからこそ把握が必要 |
| 航空 | 疲労亀裂の前兆 | 稀少な信号を見逃すと重大事故に |

代謝制御のみの場合、これらの「希少だが高価値」なノードは、ゴミデータと区別されずに削除されてしまいます。これは統計学でいう**偽陰性**（false negative）に相当します。本来保持すべきデータを「不要」と誤判定して削除する問題です。

### 2.2 検証結果

リファレンス実装の28業界シナリオで検証した結果、代謝制御のみ（保護なし）では、希少な価値データ（Truth nodes）の生存率が著しく低くなります。

| シナリオ | 保護なし | 保護あり |
|---|---|---|
| 金融（不正信号） | 0/6 (全滅) | 保護済み |
| 通信（バックアップ経路） | 0/6 (全滅) | 4/6 生存 |
| サイバーセキュリティ（APT信号） | 0/6 (全滅) | 2/6 生存 |
| スマートグリッド（障害前兆） | 0/6 (全滅) | 6/6 生存 |
| EC（ニッチ商品） | 部分消失 | 保護済み |
| 医療（希少疾患） | 部分消失 | 保護済み |

**28業界の全てで、保護なしでは希少データが消失し、保護ありでは生存率が改善されました。**

## 3. 二段階審査メカニズム

### 3.1 状態遷移図

希少性保護は、ノードに対する二段階の審査期間を設けます。

```
    degree <= 1 のノードを検出
              |
              v
    ┌────────────────────────┐
    │  Phase 1: 無条件保護    │  Twait1 ステップ（推奨: 50）
    │  代謝制御の対象外       │
    │  整合性発見が接続探索   │
    └───────────┬────────────┘
                |
         ┌──────┴──────┐
         |             |
    接続獲得        接続なし
         |             |
    保護終了     ┌─────────────────────────┐
    (正常復帰)   │  Phase 2: 条件付き猶予    │  Twait2 ステップ（推奨: 50）
                 │  最終判定期間             │
                 └──────────┬──────────────┘
                            |
                     ┌──────┴──────┐
                     |             |
                接続獲得 or     接続なし &
                degree > 0     degree = 0
                     |             |
                保護終了         削除
                (正常復帰)
```

### 3.2 各フェーズの意味

**Phase 1（無条件保護）**

ノードが希少候補として検出されると、まずPhase 1に入ります。この期間中、ノードのエッジは代謝制御の対象外となり、減衰しません。同時に、整合性発見（第4回で解説）がこのノードと他のノードの間に隠れた関連がないか探索します。

Phase 1の目的は「まだ接続が見つかるかもしれない」という猶予期間を与えることです。

**Phase 2（条件付き猶予）**

Phase 1を経過しても接続が見つからなかった場合、Phase 2に移行します。Phase 2ではノードの階層がRareからCoreに変更され、通常のノードと同じ管理下に置かれます。

Phase 2終了時に、以下の条件を**両方**満たすノードのみが削除されます。

- 接続獲得フラグ（spoke_up）が偽
- 次数（degree）が 0

つまり、Phase 1 + Phase 2の合計100ステップの間に一度でも新しい接続が生まれれば、ノードは生き残ります。

### 3.3 パラメータ

| パラメータ | 記号 | 推奨値 | 範囲 |
|---|---|---|---|
| Phase 1 期間 | Twait1 | 50 | 30-70 |
| Phase 2 期間 | Twait2 | 50 | 30-70 |
| 希少判定閾値 | rare_degree_max | 1 | 0以上 |

Twait1 = Twait2 = 50は、実証評価（図13B）で推奨されている値です。合計100ステップの審査期間は、整合性発見が隠れた関連を探索するのに十分な時間を確保します。

## 4. 保護集合と代謝制御の連携

希少性保護と代謝制御がどのように連携するかを整理します。

```
代謝制御: エッジ(u, v) を処理
  |
  v
ノードu または ノードv が保護集合に含まれるか？
  |
  ├── はい → このエッジはスキップ（減衰しない）
  |
  └── いいえ → 通常通り減衰処理
```

保護集合（protected set）に含まれるノードに接続するエッジは、代謝制御の対象外です。これにより、希少ノードの既存の接続が保護期間中に消失することを防ぎます。

重要なのは、**保護はゴミの除去を妨げない**ということです。保護されるのは希少ノードに接続するエッジのみであり、グラフの他の部分では代謝制御が通常通り動作します。

28業界の検証で、保護ありでもゴミノードは正しく除去されることが確認されています。

## 5. 実装

### 5.1 希少候補の特定

```python
class RarityProtection:
    def identify_rare(self, graph: Graph, t: float) -> list[int]:
        """次数が rare_degree_max 以下のノードを希少候補として特定"""
        return [
            nid
            for nid, node in graph.nodes.items()
            if node.phase == "normal"
               and graph.degree(nid) <= self.rare_degree_max
        ]
```

`phase == "normal"` の条件により、既に保護中のノードが重複して検出されることを防ぎます。

### 5.2 保護の開始

```python
    def enter_protection(self, graph: Graph, nid: int, t: float) -> None:
        """ノードをPhase 1保護に入れる"""
        node = graph.nodes[nid]
        node.is_protected = True      # 代謝制御の対象外に
        node.phase = "phase1"          # Phase 1を開始
        node.phase_start_time = t      # 開始時刻を記録
        node.spoke_up = False          # 接続獲得フラグを初期化
        node.layer = "rare"            # Rare層に配置
        self.protected.add(nid)        # 保護集合に追加
```

### 5.3 フェーズの更新

```python
    def update_phases(self, graph: Graph, t: float) -> dict[str, int]:
        """全保護ノードのフェーズを更新"""
        released = 0
        removed = 0

        for nid in list(self.protected):
            node = graph.nodes[nid]
            elapsed = t - node.phase_start_time

            if node.phase == "phase1" and elapsed >= self.twait1:
                # Phase 1 終了 → Phase 2 へ
                node.phase = "phase2"
                node.phase_start_time = t
                node.layer = "core"  # Core層へ移行

            elif node.phase == "phase2" and elapsed >= self.twait2:
                if not node.spoke_up and graph.degree(nid) == 0:
                    # 接続なし & degree 0 → 削除
                    graph.remove_node(nid)
                    self.protected.discard(nid)
                    removed += 1
                else:
                    # 接続獲得 or degree > 0 → 保護解除、正常復帰
                    node.phase = "normal"
                    node.is_protected = False
                    self.protected.discard(nid)
                    released += 1

        return {"released": released, "removed": removed}
```

## 7. 設計上の選択とその根拠

### 7.1 なぜ二段階なのか

一段階の審査（例: 50ステップの猶予のみ）では不十分です。

- **猶予が短すぎると**: 接続の遅い「スロースターター」ノードを殺してしまう
- **猶予が長すぎると**: 本当のゴミが長期間残留し、メモリを圧迫する

二段階にすることで:
- Phase 1: 「まだチャンスがある」期間（整合性発見が接続を探す）
- Phase 2: 「最後の猶予」期間（本当に不要かの最終確認）

これにより、スロースターターには十分な時間を与えつつ、本当のゴミは合計100ステップ後に確実に除去できます。

### 7.2 なぜ degree = 0 が削除条件なのか

Phase 2終了時の削除条件は `not spoke_up AND degree == 0` です。degree > 0（まだ何らかの接続がある）ノードは、たとえspoke_upがFalseでも削除しません。

これは「疑わしきは罰せず」の原則です。接続が1本でも残っているなら、そのノードはまだグラフの一部として機能している可能性があります。

## 8. まとめ

希少性保護の要点をまとめます。

| 特性 | 内容 |
|---|---|
| **核心** | 低接続 ≠ 低価値。即座の削除を防ぐ二段階審査 |
| **Phase 1** | 無条件保護（Twait1ステップ）、代謝制御の対象外 |
| **Phase 2** | 条件付き猶予（Twait2ステップ）、最終判定 |
| **削除条件** | 接続獲得なし AND degree = 0 |
| **実証** | 28業界全てで保護による生存率改善を確認 |

しかし、保護するだけでは受動的です。保護期間中に能動的に隠れた接続を探す仕組みが必要です。次回は、ラプラシアン固有値を使った**整合性発見**のメカニズムを解説します。
